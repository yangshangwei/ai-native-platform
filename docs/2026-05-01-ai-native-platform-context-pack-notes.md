# AI Native 云开发平台：需求阶段 Context Pack 讨论记录

日期：2026-05-01

## 15. 需求阶段如何感知工程背景

用户提出产品需求时，人通常天然知道工程背景：已有模块、历史实现、技术债、约束、曾经踩过的坑。AI 如果只看用户一句需求，很容易脱离工程现实。

推荐策略不是“提前完整梳理”或“开发时才第一次梳理”二选一，而是：

> 薄初始化 + 需求阶段按需感知 + 开发/验收后回写完善。

也就是：

```text
不是一上来把整个项目文档化
也不是每次需求都从零读代码
而是先建一层很薄的项目地图
之后每个需求触发一次定向上下文收集
做完后把稳定结论回写成长期资产
```

### 为什么不只靠提前梳理

提前完整梳理成本高，且容易过期：

- 很多项目没有完整文档
- 用户没有耐心先做很重的 onboarding
- AI 生成的大而全文档，人不一定看
- 具体需求真正需要的上下文通常只占项目一小块

### 为什么不只靠临时梳理

每次需求都临时读代码也不稳定：

- AI 可能只读到局部实现
- 不知道历史上为什么这么做
- 容易误把旧代码当最佳实践
- 每次都重复理解项目，成本高

## 16. 三层上下文模型

### 第一层：项目薄地图

初始化时只生成低成本项目地图，不追求完整。

内容包括：

```text
1. 项目是什么
2. 技术栈是什么
3. 如何运行 / 测试 / 构建
4. 主要模块有哪些
5. 数据模型 / API / 路由大概在哪
6. 已知约束和危险区
7. 已有文档在哪里
```

目标不是写“架构圣经”，而是让 AI 知道下次该去哪儿找东西。

### 第二层：需求阶段 Context Pack

用户提出需求后，平台先生成当前需求相关的上下文包，不急着写 PRD 或方案。

例如用户说“我想加一个团队权限系统”，平台自动查：

```text
1. 已有 requirement：有没有用户、团队、角色、权限相关能力
2. architecture：认证、用户模型、API 中间件在哪里
3. 代码：已有 user / team / org / role / permission 命名
4. 历史 feature：之前有没有类似权限设计
5. issue：权限相关 bug 有没有历史坑
6. decision：有没有权限模型相关长期约束
```

然后给人简洁结果：

```text
我扫到这些相关上下文：

- 当前已有 User / Team 模型，但没有 Role 表
- API 层已有 auth middleware，可以复用
- 前端已有 team settings 页面
- 历史决策：不要在前端做权限判断，后端是权威
- 已知坑：之前 workspaceId 和 teamId 混用导致过 bug

基于这些，我需要确认 3 个问题：
1. 权限粒度是页面级、操作级，还是资源级？
2. 是否需要自定义角色？
3. 是否需要审计日志？
```

### 第三层：验收后回写长期资产

需求做完、方案确认、代码落地、验收结束后，把稳定结论回写：

```text
requirements/：这个能力为什么存在
architecture/：系统现在怎么实现
features/：这次 feature 的设计和验收
compound/：踩坑、技巧、决策
```

这样下一次需求阶段，Context Pack 会更准。

## 17. 低门槛但高质量的做法

核心原则：不要让用户手动填一堆文档。

用户只输入一句话：

```text
我想加一个团队权限系统
```

平台自动补上下文：

```text
我已检查：
- 当前代码模型
- 已有架构文档
- 历史 feature
- 历史 bug
- 技术决策
```

然后只问 3-5 个真正影响方案的问题。

生成的需求草稿应包含：

```text
需求目标
现有工程背景
用户故事
不做什么
成功标准
相关模块
风险点
待确认问题
```

Web UI 可做成左右结构：

```text
左侧：需求草稿
右侧：相关工程背景
  - 相关模块
  - 已有实现
  - 历史决策
  - 相关 bug
  - 可复用代码
  - 风险提示
```

每条工程背景都要有证据链接，例如：

```text
auth middleware: src/server/auth.ts
Team model: prisma/schema.prisma
历史决策: codestable/compound/2026-04-xx-decision-auth.md
```

高质量不是 AI 说“我觉得”，而是每个关键判断都有证据来源。

## 18. 推荐比例

推荐把上下文工作量分配为：

```text
初始化：20% 梳理
需求阶段：60% 定向上下文收集
验收后：20% 回写沉淀
```

不要在初始化阶段做 100%。

## 19. 例子：导出 CSV

用户说：

```text
我要做导出 CSV。
```

平台不应该直接问“你要导出哪些字段？”，而应先查工程背景：

```text
- 项目已有 ExportButton 组件
- 后端已有 /api/reports endpoint
- 数据表里金额字段用 cents 存储
- 历史 decision 规定：导出功能必须走异步任务，不能同步阻塞请求
- 之前 issue 记录过：大文件导出导致过超时
```

然后问更高质量的问题：

```text
我看到项目已有导出相关约束：大文件必须异步导出，金额字段以 cents 存储。

请确认：
1. 这次 CSV 是小数据即时下载，还是也必须走异步导出？
2. 金额字段展示为 cents 还是格式化后的金额？
3. 是否需要复用已有 ExportButton？
```

## 20. 本质结论

要解决的不是“AI 会不会写需求”，而是：

> AI 在写需求时有没有项目记忆和工程现实感。

平台应该默认在每个 PRD / Design 之前生成轻量 Context Pack。

Context Pack 要满足：

- 自动生成
- 可折叠
- 有证据链接
- 只展示和当前需求相关的内容
- 不要求用户手写
- 不要求项目一开始文档完整

一句话：

> 低门槛靠自动感知，高质量靠证据约束，长期质量靠持续回写。

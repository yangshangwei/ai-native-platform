## 4. Specific design questions the brainstorm should resolve

These are the open Q's for `trellis-brainstorm` to walk the user through. Recommended first pass:

### Q1: WorkflowStage enum extension — naming + count

Wave 2 roadmap PRD lists candidate names: `report` / `analyze` / `scan` / `apply`. But:
- Does `issue.standard` need its own `fix` / `fix_note` stage, or does it reuse `implementation`? (cs-issue-fix skill semantically maps to implementation but with a different prompt template.)
- Does `refactor.standard` use `apply` (vs `implementation`)? Or fold into `implementation`?
- Should we add a `verify` stage distinct from `build_test` for "did the fix actually fix the bug" semantics?

Recommend converging on **smallest enum extension that's honest about the work shape**.

### Q2: WorkflowRunType extension

Add `'issue' | 'refactor'`? Or keep `'feature' | 'bugfix' | 'smoke'` and use FlowId for variation?

`bugfix` already exists — could `issue.standard` map onto `bugfix` runType? The handoff doesn't specify.

### Q3: Gate engine extensions

`acceptance_gate` is currently traceability-driven (assumes requirement/design artifacts exist). Issue / refactor flows likely don't have those. Options:
- Add per-flow gates: `issue_acceptance_gate` / `refactor_acceptance_gate`
- Generalize `acceptance_gate` to handle absent artifacts gracefully (W2-3 also has this caveat)
- Skip acceptance entirely on these flows and rely on `review` agent + human approval

### Q4: skillId activation gate

W2-1 declared kind/skillId as placeholders. W2-2 introduces 2-3× more agent stages each with skillIds. Continue placeholder (every executeXxx hard-codes its skill via `mustSkill(stage)`)? Or wire `step.skillId` into a generic `executeAgentStep(step, ctx)` and retire most executeXxx?

This is the biggest architecture decision in W2-2. **Recommendation**: continue placeholder for W2-2, activate at W2-4 — W2-2 is the second extensibility test of Wave 2 abstractions, not the right place to refactor them.

### Q5: New stage executors

Each new stage needs an `executeXxx` (or generic equivalent if Q4 activated). For each:
- What does it write (artifact kind)?
- What gates run?
- Does it pause for human?
- Is it skill-driven (agent) or engine?

### Q6: skill specs

Backend `findSkillForStage(stage)` returns a `SkillSpec`. Currently only feature stages have specs. W2-2 needs specs for: `report` / `analyze` / `scan` / `apply` (or whatever Q1 lands on). Are there existing CodeStable skill files (`cs-issue-report` etc.) that need ingestion, or do we draft fresh SkillSpec stubs?

### Q7: PR breakdown

Wave 2 roadmap says ~5 PRs. Suggested cuts:
- PR1: shared types — extend WorkflowStage + WorkflowRunType (if Q2)
- PR2: gates + executeXxx for new stages
- PR3: FLOW_REGISTRY entries for `issue.standard` + `refactor.standard`
- PR4: route + CLI plumbing (KNOWN_FLOW_IDS update + tests)
- PR5: spec doc update + skill stubs

## 7. Open questions / risks for the new session

- **Acceptance gate semantics on non-feature flows** — already a known W2-3 caveat (acceptance_gate's traceability check assumes requirement/design exist). W2-2 is when this becomes acute (issue/refactor don't have those artifacts at all). Q3 above.
- **Knowledge promotion artifact-kind expansion** — `promoteAcceptedDraftToKnowledge` is hard-coded for `'requirement_draft' | 'design_doc'`. New flows producing `analysis_doc` / `refactor_plan` etc. either need this expanded or need a different promotion path.
- **WorkflowStage enum bloat** — every new value is a `dispatchStep` switch case + a `runStage`/executeXxx + a possible gate. Resist adding "convenience" stages (e.g. don't add both `analyze` AND `triage` if one suffices).
- **issue vs bugfix overlap** — `WorkflowRunType` already has `'bugfix'` (unused in orchestrator). Issue work is bug work. Decide whether to:
  - Keep `'bugfix'` for backward-compat and add `'issue'` as alias
  - Repurpose `'bugfix'` as the runType for `issue.standard`
  - Delete unused `'bugfix'` and just use `'issue'`

Pick consciously; it shows up in DB rows.

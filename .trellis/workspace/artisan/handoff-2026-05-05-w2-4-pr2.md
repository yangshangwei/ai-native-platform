### Key gotchas

- **Migration is idempotent**: see `apps/api/src/store/db.ts:285-330` for the W2-1 PR3 `flow_id` migration as template. Pattern:
  ```ts
  if (!columnNames(db, 'workflow_runs').includes('start_stage')) {
    db.run("ALTER TABLE workflow_runs ADD COLUMN start_stage TEXT");
  }
  ```
- **Nullable column**: NO `DEFAULT 'init'` or similar — null means "start from head". Existing rows pre-W2-4 will have NULL `start_stage` (correct semantically).
- **`runWorkflow()` slicing**: do NOT branch by flowId in dispatchStep; pure positional slice from the index lookup. dispatchStep's `_exhaustive: never` invariant must remain intact (W2-1 ADR Q1=α).
- **Skipped stages' inputs**: skipped stages don't run, so their outputs (e.g. `inputs.requirement.md`) are not populated. Downstream skills must handle this gracefully — pattern already established in W2-3 fastforward (skill.implementation.inputs.design.required = false; cascade fallback in instructions). For PR2, no further skill changes are required since the auto-pick auto-fills startStage with router's recommendation that respects flow shape.
- **`run.startStage` validation site**: orchestrator. Don't validate inside `createWorkflowRun()` — the run might be created with a startStage that's later corrected via UI override. Orchestrator throws on dispatch.

## 5. PR3 — concrete implementation plan

**Goal**: server-side `POST /router/recommend` endpoint + `createWorkflowRun` calls `recommend()` when `body.flowId` is missing. PRD R15-R17 + R27 + AC-12 + AC-13 + AC-14 + AC-15.

### Files to touch

| File | Change |
|---|---|
| `apps/api/src/routes/router.ts` NEW | Hono route module: `POST /router/recommend` body: `RouterInput` JSON → 200 with `RouterRecommendation` JSON. Validate `projectId` (lookup `store.projects.get`) and `runType` (must be a known WorkflowRunType); 400 on invalid. Pure read; no DB writes. |
| `apps/api/src/app.ts` | Mount `/router` route via `app.route('/router', routerRoutes)` (mirroring how `/workflow-runs` is mounted). |
| `apps/api/src/workflow-engine.ts` | `createWorkflowRun()` integration: if `params.flowId === undefined`, call `recommend({ projectId, title, runType: params.type })`; populate `flowId = rec.flowId`, `startStage = rec.startStage`. Add audit log payload field `routerRecommendation: { flowId, startStage, rulesFired }` to the existing `workflow_run.created` audit entry. |
| `apps/api/test/router-route.test.ts` NEW | Endpoint integration tests (mirror W2-3 / W2-2a style with `app.request('/router/recommend', ...)`). Cases: 200 with valid body; 400 missing projectId; 400 unknown projectId; 400 unknown runType; smoke type → feature.standard recommendation. |
| `apps/api/test/workflow-engine.test.ts` | Tests: (a) createWorkflowRun without flowId → router auto-fills (e.g. type='bugfix' → flowId='issue.standard'); (b) createWorkflowRun with explicit flowId → router skipped (verify via spy or by setting up a project that would recommend differently); (c) audit log contains `routerRecommendation` payload. |

### Key gotchas

- **`POST /router/recommend` is read-only**: NO `db.run(...)` or `store.X.set(...)` calls. The endpoint is pure compute. Sole-writer principle for `workflow_runs` is preserved (createWorkflowRun stays the only writer).
- **`projectId` validation**: `store.projects.get(projectId)` must return non-null. Some tests fixtures register projects via `storeMod.store.projects.set(...)` directly; reuse `registerProject()` helper from existing route tests.
- **`runType` validation**: cast through `KNOWN_RUN_TYPES` set check (similar to KNOWN_FLOW_IDS). Reject unknown to prevent typos creating invalid recommendations.
- **`createWorkflowRun` recursion**: do NOT recurse — if `params.flowId === undefined`, call `recommend()` (synchronous), then proceed with the run creation. Don't call `createWorkflowRun` again with the filled flowId.
- **Audit payload extension**: existing `audit(id, 'workflow_run.created', { type, title, sourceBranch, flowId })` call — add `routerRecommendation` field when router was used. When user supplied flowId explicitly, omit the field (or set to null) so audit log shows the override.
- **Test isolation**: each test should use unique `projectId` to avoid bleed-over via `store.knowledgeArtifacts.byProject` (which can accumulate across tests in the same SQLite db).
